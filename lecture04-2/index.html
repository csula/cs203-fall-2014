Introduction

Developing software these days often amounts to re-using or building on top of existing software. These existing software are often modelled or described as framework.

How do framework and library differ?

  • A library solves a specific task or a set of specific tasks.
  • A framework enforces how an application or a set of applications behave and interact.

Generally speaking, when writing software we try to build software on top of a framework. The implication here is that the framework developer are "super smart" when in fact that might not always be the case.

Good Framework: should be simple and does not try to do too many things.

So how do we look at data structure lessons (in this class) as a framework design?

  • We are not building a library for road and communication network
  • We are building both an application and a framework for future applications
  • Sometimes thinking framework is better than thinking application

Bottom Line: A framework is extensible where as a library is restrictive.

Revisit Object-Oriented Programming

One of the core tenet's of object-oriented programming is encapsulation, in other words, data- and action-hiding. For our project so far, we've introduced a few clases:

  • Point
  • Circle

So far, a Point and a Circle do not convey information about a road. So how does one connect the dots (no puns intended)?

We can make an argument that a road is consisted of a set of points. We can also say that a road is consist of a set of segments and a segment is consisted of two (or more) points. What then is an intersection?

Design a Road class, a segment class, and an Intersection class. Your classes should utilize the containers (collection) that we've discussed in class.

In addition to designing the structure and behavior of the ojects, we need to look at how they interact. What are the relationships between these classes?

Framework Building

Is there a method (or madness) to how one builds a framework?

Yes! Start with something (anything). Trials and errors. It should be noted that no one sits down and say, "I am going to build a framework today and it will great!" Iteratively refined as we move forward.

Let's recap:

  1. Design Road, Segment, and Intersection
  2. Utilize container/collections
  3. Define basic relationship between classes
  4. Application flow (this is the next step)

Note that step #4 above suggests that we are working toward one application. This is perhaps the biggest mistake with framework development. We are not working toward one application -- we're working toward many applications.

So what are these applications?

  1. Data acquisition: downloading and acquiring map and road network data
  2. Infrastructure: we need to build our road network
  3. Visualization: we need to illustrate (visually) our model
  4. Computation: we need to run our model to computations of some flavor
